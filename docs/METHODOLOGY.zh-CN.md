# 两仪 · Liangyi — AI 驱动的个人产品开发工作流 v1.0

[English](Liangyi.md)

> **太极生两仪** ——《易经·系辞》
> 一个统一的原始构想（太极）生发出对立的阴阳两极（两仪），
> 对立的张力塑造所有更丰富的结构。

> 一套融合多模型交叉验证、对抗式专家角色博弈、决策日志、Skills 调度的标准化 AI 辅助产品开发框架。
> 核心理念：通过 Best Minds 角色分配制造结构性冲突，模拟真实团队中不同职能角色的博弈。
> 适用场景：个人开发者 / AI 产品工程师，从 0 到 1 构建 Web 产品。
>
> **模型兼容性：** 本工作流运行在 Claude Code 中（模型 A = Claude）。对抗模型（模型 B）推荐使用 Gemini，但 GPT / Qwen / 豆包 / DeepSeek 等均可。关键是使用**不同厂商的模型**以获得最大的训练偏见差异。本文档中"Gemini"均指代你选择的模型 B。
>
> 本工作流已封装为 [Liangyi Skill 家族](https://github.com/wenboxia/liangyi)，可通过 `npx skills add` 一键安装。

### Skills 调度矩阵（速查表）

| 工作流阶段 | 推荐 Skill | 使用方式 |
|-----------|-----------|---------|
| Phase 0 Idea 精炼 | — | 无需 Skill，纯 prompt 操作 |
| Phase 1 角色选择 | **Best Minds** | 选择有结构性冲突的专家角色对，记入决策日志 |
| Phase 1 Brainstorm | **Best Minds** + **Product Manager Skills** | Claude 和 Gemini 分别以不同专家身份出方案；PM Skills 构建产品框架 |
| Phase 1 知识验证 | **NotebookLM Skill** | 查询你自己的知识库，用真实数据验证假设 |
| Phase 2 交叉验证 | **Best Minds** | Claude 切换投资人角色批判；趋同时触发"拆台专家"压力测试 |
| Phase 3 PRD 生成 | **Product Manager Skills** | 直接驱动 PRD、用户故事、需求文档的生成 |
| Phase 3 PRD 校验 | **Best Minds** | 以目标用户身份阅读 PRD，检验产品表达是否跑偏 |
| Phase 3 技术文档 | **Everything Claude Code** | 参考其 agents/hooks/commands 配置模板 |
| Phase 4 开发执行 | **Superpowers** | 全程使用：/plan 规划、/review 审查、/loop 迭代、/batch 批处理 |
| Phase 4 技术决策 | **Best Minds** | 遇到选型纠结时即时召唤领域专家，不接受"两者各有优劣" |
| Phase 4 能力扩展 | **Find Skills** | 遇到新需求时自动发现和推荐可用 Skill |

**使用原则：**
- Phase 1（Brainstorm）：将 Skills 清单作为上下文附给 Claude 和 Gemini，让它们在设计方案时考虑"有哪些能力可以用"，避免方案超出实际能力或低估可用能力
- Phase 3（文档生成）：Skills 清单直接影响技术选型和开发指南，必须作为输入
- Phase 4（开发执行）：`dev-guide.md` 中会标注每个步骤推荐使用的具体 Skill
- **Skills 组合原则**：单个任务优先用一个 Skill；复杂任务可组合，但注意避免指令冲突（如两个 Skill 对同一行为有矛盾要求时，以更专业的 Skill 为准）

## Phase 0｜Idea 捕获

**输入：** 你脑中的一个粗糙想法（一句话即可）

**操作：**

1. 用语音备忘录或随手记录，不要过滤，保留最原始的表述。
2. 将原始想法发送给 **Gemini**，使用以下 prompt：

```
你是一个 prompt 精炼专家。我会给你一段口语化的产品想法描述，请你：
1. 去掉口语化表述、重复和填充词
2. 只做语言层面的精炼，不要改变我的原始意图和方向
3. 保留我提到的所有具体细节和约束条件
4. 输出一段结构清晰、表达精准的想法描述（200字以内）

我的原始想法：[粘贴你的原始描述]
```

**输出：** 一段精炼后的 idea 描述

## Phase 1｜对抗式 Brainstorm（核心创新环节）

> 💡 **关键变化：不再是串行评审，而是让两个模型以不同专家身份独立生成完整方案，互不可见。**
> 
> **Best Minds 深度集成：** 通过给两个模型分配有结构性冲突的专家角色，制造的不是"两个 AI 的随机差异"，而是"两种真实职业立场的系统性博弈"。

### Step 1A-0 — 专家角色选择（先于方案生成）

在发送 idea 之前，先根据你的产品方向选择两个**核心关注点有张力**的专家角色：

```
推荐角色对（根据产品类型选择）：

支付/金融类产品 → 安全架构师 vs 增长黑客
社交/内容类产品 → 用户体验设计师 vs 数据变现专家
工具/效率类产品 → 极简主义产品经理 vs 功能全栈架构师
AI/技术类产品  → AI 研究员 vs 商业化运营总监
教育类产品     → 教育心理学家 vs 游戏化设计师

通用高张力组合：
- 投资人（关注ROI和规模） vs 一线工程师（关注可行性和维护成本）
- 行业老炮（关注壁垒和经验） vs 跨界新人（关注颠覆和第一性原理）
- 目标用户代言人（关注痛点） vs 竞品分析师（关注差异化）
```

**选择原则：** 两个角色的优先级排序要天然冲突。如果两个角色会给出相同建议，说明你选的张力不够，换一对。

在 `decision-log.md` 中记录你的角色选择：

```markdown
### 决策点 0：专家角色分配
- Claude 扮演的专家：[角色名 + 为什么选这个角色]
- Gemini 扮演的专家：[角色名 + 为什么选这个角色]
- **选择这对角色的理由：** [它们的冲突点在哪里？这个冲突为什么对本项目有价值？]
```

### Step 1A — Claude 独立方案（以指定专家身份）

将精炼后的 idea 发送给 **Claude**（Claude Code 的 CoWork 模式），使用以下 prompt：

```
【Best Minds 模式激活】

请以[专家角色A]的身份思考以下产品想法。你拥有这个领域 20 年的经验，
你的思考方式、优先级排序、风险偏好都应该完全符合这个角色。

关键要求：
- 如果你作为[角色A]的判断与"通用产品经理"的建议有冲突，请优先遵循你的角色立场
- 明确标注哪些建议是"出于[角色A]的立场才会这样说"
- 在方案中体现这个角色独有的认知偏好和盲点（是的，故意保留偏见）

基于以上角色身份，请你帮我：

1. 以你的专业视角分析这个想法的核心价值主张
2. 从你最关注的维度评估可行性
3. 指出你这个角色最担心的致命缺陷
4. 提出你认为更好的替代方案或改进方向（至少 2 个）
5. 给出一个你推荐的最终方案雏形

在整个过程中，主动向我提问来明确需求。不要假设，不确定的地方就问我。

同时，以下是我当前可用的 Skills 工具集，请在设计方案时考虑如何利用这些能力，并指出哪些环节应该调用哪个 Skill：
[粘贴你的 Skills 注册表]

我的想法：[粘贴精炼后的描述]
```

与 Claude 进行多轮上下文对话，直到得到一个**完整的方案 A**。

### Step 1B — Gemini 独立方案（以另一个专家身份，同步进行）

**关键：不要把 Claude 的任何输出给 Gemini 看。** 用同样的精炼 idea，但指定**不同的专家角色**：

```
【Best Minds 模式】

请以[专家角色B]的身份思考以下产品想法。你拥有这个领域 20 年的经验，
你的思考方式、优先级排序、风险偏好都应该完全符合这个角色。

关键要求：
- 你不需要面面俱到，只需要从你的专业角度给出最诚实的判断
- 如果你作为[角色B]认为某些方面不重要，可以直接说"这不是我关注的重点"
- 明确标注你的角色立场如何影响了你的建议

基于以上角色身份，请你：

1. 从你最关注的维度出发，重新审视这个想法是否瞄准了真实需求
2. 给出你认为最佳的产品形态和核心功能定义
3. 指出你这个角色视角下最大的 3 个风险
4. 提供一个完整的方案建议

注意：请完全基于你自己的角色判断，给出独立的方案。

以下是我当前可用的工具和 Skills，请在方案中考虑这些能力的适用性：
[粘贴你的 Skills 注册表]

我的想法：[粘贴同样的精炼描述]
```

**输出：** 两份带有明确立场差异的独立方案（方案 A = 角色A视角，方案 B = 角色B视角）

### Step 1C — 人工对比 + 决策日志

> 💡 **这是整个流程中你作为"人"最不可替代的环节。**

打开一个新文档，命名为 `decision-log.md`，记录：

```markdown
## Decision Log

### 决策点 1：产品核心方向
- [角色A]方案主张：[一句话 + 这个主张背后的角色立场]
- [角色B]方案主张：[一句话 + 这个主张背后的角色立场]
- **两个角色的核心分歧点：** [它们在什么问题上给出了截然不同的答案？]
- **我的选择：** [选了哪个 / 如何融合]
- **选择理由：** [为什么？基于什么判断？两个专家都没考虑到什么？]

### 决策点 2：[下一个分歧点]
...
```

**每个决策点必须回答：** "我为什么选 A 不选 B？这个判断 AI 做不了，因为______。"

## Phase 2｜交叉验证与融合

### Step 2A — Claude 批判（切换新角色）

将你融合后的方案发回 Claude，**但此时让 Claude 切换到一个新的专家角色**，不要沿用 Phase 1 的角色：

```
【Best Minds 模式 — 角色切换】

请放下你之前的[角色A]身份。现在请以一个对这个 idea 持怀疑态度的资深投资人身份审视以下方案。

你看过太多失败的项目，你的本能是找出每个方案最可能死掉的原因。请你：

1. 指出你认为最薄弱的 3 个环节
2. 对每个薄弱环节给出具体的加固建议
3. 判断这个方案的"一号死因"最可能是什么
4. 最终输出一版改进后的完整方案

方案内容：[粘贴你的融合方案]
```

### Step 2B — Gemini 单盲审查（不给原始 idea）

> 💡 **这是你原始工作流中最有价值的洞察：让评审者不知道创造者的意图，只评价方案本身。**
> 
> 知道原始 idea 的评审者会不自觉地"帮你补脑"——你没写清楚的地方它会用你的 idea 去填充。不知道 idea 的评审者不会补脑，它评价的是"这份方案本身是否逻辑自洽、是否有说服力"。

**关键操作：开一个全新的 Gemini 对话窗口。** 不要用 Phase 1 中已经知道你 idea 的那个 Gemini 会话。

```
以下是一份产品方案。我没有额外的背景信息给你，请你纯粹基于这份方案本身来评价。

请你：

1. 用一句话总结你认为这个方案要做什么（不要猜测背后的意图，只说方案本身传达了什么）
2. 这份方案的逻辑是否自洽？有没有"读不通"的地方？
3. 如果你是一个不了解背景的决策者，你会批准这个方案吗？为什么？
4. 指出方案中表述模糊、有歧义、或需要额外解释才能理解的部分
5. 给出你的改进建议

方案内容：[只粘贴 Claude 改进后的方案，不附带任何你的 idea 原文、背景、动机]
```

**校验方法：**
- Gemini 第 1 题的一句话总结 vs 你的原始意图 → 如果偏差大，说明方案的表达有问题，而不是 Gemini 理解力的问题
- Gemini 指出的"读不通"的地方 → 这些是你和 Claude 因为共享上下文而忽略的"知识诅咒"盲区

在 `decision-log.md` 中记录：
```markdown
### 单盲审查校验
- Gemini 的一句话理解：[记录]
- 与我原始意图的偏差：[具体在哪里]
- Gemini 指出的"读不通"之处：[列出]
- **我的判断：** [这些偏差说明方案表达有问题 / 说明方案方向有问题 / 确认无偏差]
```

### Step 2C — Gemini 知情复审（可选）

如果 Step 2B 的单盲审查发现了显著偏差，你修正方案后，可以**在 Phase 1 的那个 Gemini 会话中**（它知道你的 idea）再做一轮知情审查，确认修正后的方案既表达清晰又符合你的原始意图。

```
之前的方案经过外部评审后做了一些调整。
你之前了解我的原始想法，请对比评估：
1. 修改后的方案是否仍然对齐我最初的核心目标？
2. 为了表达清晰而做的调整是否牺牲了什么重要的东西？
3. 还有需要进一步改进的地方吗？

修改后的方案：[粘贴修正后的方案]
```

### Step 2D — 拆台专家（当 Claude 和 Gemini 意见趋同时触发）

> 💡 **趋同是危险信号。** 两个模型都说"没问题"，可能是真的没问题，也可能是它们共享相似的训练偏见。此时引入第三个专家身份进行压力测试。

**触发条件：** Claude 和 Gemini 对方案的核心判断一致，没有产生实质性分歧。

**复杂度门槛：** 仅在涉及核心架构决策、战略方向或重大资源投入时触发。对于 UI 调整、文案修改或小范围功能迭代，直接跳过此步。

将方案发给 Claude（或 Gemini），使用以下 prompt：

```
【Best Minds 模式 — 拆台专家】

你现在是[这个产品领域最知名的怀疑论者/批评家]。
你的职业习惯是找出别人看不到的致命漏洞。你对"大家都觉得没问题"的共识天然警惕。

以下方案已经经过两轮独立评审，所有评审者都认为方案基本可行。
这恰恰是你最警觉的时刻。请你：

1. 找出这个方案最可能在 6 个月后被证明是错误的假设（至少 2 个）
2. 指出评审者可能因为"锚定效应"而忽略的替代路径
3. 如果你是竞争对手，你会用什么方式击败这个方案？
4. 给出你的"如果非要做，至少要先验证____"的建议

方案内容：[粘贴当前方案]
```

**如果拆台专家找到了实质性漏洞：** 回到 Step 2A 重新循环。
**如果拆台专家也无法找到致命问题：** 方案的可信度大幅提升，进入定稿。

**再次记录决策日志：** 对每轮批判的每条建议，记录你的判断（采纳 / 拒绝 / 部分采纳，以及理由）。

将最终决策反馈给 Claude，让 Claude 输出定稿。

**循环条件：** 如果方案仍有重大不确定性，重复 Phase 2。一般 2-3 轮即可收敛。

**输出：** `idea.md` — 最终确认的产品想法文档

## Phase 3｜文档体系生成

基于定稿的 `idea.md`，在 Claude Code 中依次生成以下文档：

### 3.1 PRD 文档

```
基于以下产品想法，帮我撰写一份完整的 PRD（产品需求文档）。

要求：
- 包含：产品概述、目标用户、核心功能列表、用户故事、信息架构、MVP 范围定义、成功指标
- 明确标注 MVP 阶段和后续迭代的功能边界
- 评估是否需要多 Agent 协作（Claude Code Teams 模式），给出建议

产品想法：[引用 idea.md 内容]
```

### 3.1.1 PRD 用户视角校验（Best Minds）

> 💡 **PRD 写着写着容易变成"我们能做什么"，而不是"用户需要什么"。这一步用 Best Minds 把视角拉回来。**

PRD 初稿生成后，**在定稿前**加一步用户模拟校验：

```
【Best Minds 模式 — 目标用户模拟】

你现在是[这个产品的典型目标用户画像，具体描述]。
你不懂技术，你不关心架构设计，你只关心这个产品能不能解决你的问题。

请你阅读以下 PRD，然后用你自己的话回答：

1. 这个产品是干什么的？（用一句最朴素的话说）
2. 它解决了你生活/工作中的什么痛点？
3. 你会用它吗？为什么？
4. 你最担心什么？什么会让你放弃使用？
5. 你觉得它缺了什么功能是你特别需要的？

PRD 内容：[粘贴 PRD 初稿]
```

**校验方法：**
- 如果模拟用户的理解和你的产品意图有偏差 → PRD 的表达有问题，需要修改
- 如果模拟用户说"不会用" → 回到 Phase 2 重新审视产品方向
- 如果模拟用户提出了你没想到的需求 → 评估是否纳入 MVP

在 `decision-log.md` 中记录（标记为 **AI 模拟校验结果**，非人类决策记录）：
```markdown
### [REVIEW] PRD 用户视角校验
- 模拟用户的一句话理解：[记录]
- 与我原始意图的偏差：[有/无，具体在哪里]
- **处理方式：** [修改PRD / 调整方向 / 确认无偏差]

> 注意：此条目为 AI 模拟结果，非人类决策。人类决策记录使用 [PRODUCT] / [TECHNICAL] 标签。
```

### 3.2 技术架构文档（tech-spec.md）

```
基于以下 PRD，生成一份技术架构文档，供后续 Claude Code 开发时读取。

要求：
- 技术栈选型及理由
- 项目目录结构
- 核心模块划分和接口定义
- 是否使用 Claude Code Teams 多 Agent 模式，如何分配 Agent 职责
- 开发阶段划分（建议按功能模块拆分为可独立验证的阶段）
- 关键技术风险点及应对方案
- **Skills 调度映射表**：列出每个开发阶段推荐使用的 Skill，格式如下：
  | 阶段 | 任务 | 推荐 Skill | 理由 |
  |------|------|-----------|------|
  | 示例 | UI 组件开发 | Superpower Skills | 复杂交互逻辑生成 |

可用 Skills：[引用 Skills 注册表]
PRD 内容：[引用 PRD]
```

### 3.3 CLAUDE.md（全局配置文档）

```
基于以下技术架构文档，生成一份 CLAUDE.md 全局配置文件。

要求：
- 代码规范（命名、格式、注释标准）
- Git 工作流规范
- 文件组织规范
- 测试要求
- Claude Code 在开发过程中需要遵守的约束和注意事项
- 如使用 Teams 模式，各 Agent 的职责边界

技术架构：[引用 tech-spec.md]
```

### 3.4 开发操作指南（dev-guide.md）

```
基于以上所有文档，为我（AI 产品工程师 / 个人开发者）生成一份逐步操作指南。

要求：
- 按阶段拆分，每个阶段明确：做什么、用什么工具、**调用哪个 Skill**、预期输出
- 标注哪些步骤需要人工决策（对应 decision-log.md 的记录点）
- 包含每个阶段的验收标准（怎么判断这步做完了）
- 包含常见问题和应对策略
- 如果项目规模需要 Teams 模式，说明如何启动和协调多 Agent
- **Skill 使用指南**：对每个推荐使用的 Skill，说明具体的调用方式、输入格式和注意事项

参考 tech-spec.md 中的 Skills 调度映射表来编写。
```

**输出：** 五份文档

| 文档 | 用途 | 读者 |
|------|------|------|
| `idea.md` | 产品核心想法 | 你自己 |
| `prd.md` | 产品需求定义 | 你 + Claude Code |
| `tech-spec.md` | 技术架构规划 | Claude Code |
| `CLAUDE.md` | 全局开发规范 | Claude Code |
| `dev-guide.md` | 操作指南 | 你自己 |
| `decision-log.md` | 人类决策记录 | 你自己 / 面试展示 |

> 注意：`decision-log.md` 是第六份文档，贯穿全流程持续更新，不是一次性生成的。

## Phase 4｜开发执行

1. 在 Claude Code 中加载 `CLAUDE.md` 和 `tech-spec.md`
2. 按 `dev-guide.md` 的阶段划分逐步推进
3. 每完成一个阶段，在 `decision-log.md` 中记录关键决策
4. 如遇重大方向调整，回到 Phase 2 进行交叉验证

### Best Minds 即时召唤（开发阶段专用）

开发过程中遇到技术选型纠结时，不要接受 Claude Code 给你的"通用优缺点对比"，而是用 Best Minds 切换到专家模式：

```
【Best Minds 模式 — 即时技术决策】

世界上谁最懂[当前技术问题的领域]？请以那个人的身份回答：

[你的具体技术问题，比如：在这个项目中，数据库应该选 PostgreSQL 还是 MongoDB？]

要求：
- 不要给我"两者各有优劣"的平衡答案
- 以你的专业经验直接给出你的推荐，并说明"如果选另一个，你最担心的是什么"
- 带入你在实际项目中踩过的坑
```

**适用场景举例：**
- 框架选型纠结 → 召唤"前端架构师"
- 数据库设计犹豫 → 召唤"数据库架构师"
- 性能优化方向不明 → 召唤"性能工程师"
- 安全方案不确定 → 召唤"安全专家"

**注意：** 即时召唤不需要走交叉验证流程，但结论要记入 `decision-log.md`。

## 使用须知

**这套流程解决的问题：**
- 单人开发时缺少"第二双眼睛"来挑战你的假设
- AI 容易顺着你说，缺乏真正的对抗性审查
- 面试 / 汇报时无法证明"人的判断力"在哪里

**这套流程的局限：**
- 增加了前期时间投入（适合重要项目，不适合快速原型试错）
- 决策日志需要纪律性——如果你偷懒不记，Phase 1C 的价值归零
- 多模型交叉验证可能导致过度打磨 idea 而延迟动手（建议设置时间上限：Phase 0-2 总计不超过 1 天）

**何时简化流程：**
- 小功能迭代：跳过 Phase 1B（不需要对抗式方案），直接用 Claude brainstorm
- 技术验证类项目：跳过 Phase 0-2，直接从 Phase 3 开始
- 紧急项目：只保留 Phase 1C 的决策日志习惯，其余按直觉走
